# AROGYA Inventory — Architecture

## Overview

Single Spring Boot service backing the AROGYA cafe/kiosk: tracks ingredient and product stock,
auto-deducts on each recorded sale, and surfaces low-stock items with their supplier. Backend only
(no frontend, no auth in v1 — single-owner local app).

## Stack

Spring Boot **4.1.0** · Java 21 · Gradle (`./gradlew`) · PostgreSQL · Flyway (migrations on startup)
· JUnit 5 + Testcontainers · Spotless (google-java-format, runs on every edit).

> Note: the spec named "Spring Boot 3.x"; Initializr no longer serves 3.x (EOL), so the project was
> scaffolded on 4.1.0. This changed some starter / Testcontainers / `@DataJpaTest` package names but
> not the design.

## Package structure — package by feature

```
com.arogya.cafe
├── common/                      # cross-cutting: NotFoundException, ValidationException,
│                                #   GlobalExceptionHandler (RFC 7807)
├── supplier/  {controller, service, dto, entity, repository}
├── inventory/ {controller, service, dto, entity, repository}
├── menu/      {controller, service, dto, entity, repository}
├── sales/     {controller, service, dto, entity, repository}
└── receiving/ {controller, service, dto, entity, repository, client, support, config}
```

The `receiving` module integrates the external **billscan** service (FastAPI, runs as a sidecar in
the backend pod). `client/BillScanClient` (interface) + `RestBillScanClient` (RestClient impl)
isolate the HTTP call so the service is unit-testable with a stub. `support/UnitMapper` maps scanned
unit strings to the `Unit` enum (litre → ML ×1000). Stock is applied via the existing
`InventoryItem.adjust`; outcomes are persisted as `GoodsReceipt` + `GoodsReceiptLine` rows.

Each module is a vertical slice. Files that change together live together.

## Layer responsibilities

| Layer | Rule |
|---|---|
| `controller` | Thin. `@Valid` request DTOs, map path/query → service calls, set HTTP status. No logic. |
| `service` | Owns business logic and `@Transactional` boundaries. Maps entities ⇄ `dto` records. Throws domain exceptions. |
| `dto` | `record` request/response types crossing the wire. Response records hold an entity→DTO `from(...)` factory. |
| `repository` | Spring Data JPA interfaces only. Custom queries via `@Query` (e.g. `findLowStock`). |
| `entity` | JPA entities + enums. Behavior on the entity where it belongs (`deduct`, `adjust`, `isLowStock`, `replaceRecipe`). Never serialized over the wire. |

## Key decisions

- **Unified inventory** ("Option A"): one `INVENTORY_ITEM` table for raw materials and resale goods,
  distinguished by `unit` and by how menu items reference them (recipe vs `resale_item_id`).
- **Sales never blocked on low stock** — a sale physically happened, so it is always recorded;
  stock may go ≤ 0 and the low-stock list flags it.
- **Transactional deduction** — a multi-ingredient sale fully applies or fully rolls back.
- **Manual replenishment** via the adjust endpoint; reordering is an alert, not an integration
  (no purchase orders / supplier APIs in v1).
- **Constructor injection** with `final` fields; **DTOs, not entities**, over the wire; **central**
  RFC 7807 error translation.

## Persistence & migrations

Flyway owns the schema (`ddl-auto: validate`). `V1__schema.sql` creates tables; `V2__seed.sql`
(generated by `scripts/gen_seed.py` from the menu spreadsheet) seeds the catalog. Applied migrations
are immutable — changes add a new `V__` script.

## Testing strategy

- **Unit** (Mockito) — service logic: deduction (recipe + resale), adjust floor, validation.
- **Slice** (`@DataJpaTest` + Testcontainers) — repository queries incl. `findLowStock`.
- **Integration** (`@SpringBootTest` + Testcontainers) — full sale flow + rollback, low-stock,
  V1+V2 migrations apply cleanly. Lives in `src/integrationTest`, run via `./gradlew integrationTest`.

## Running locally

`./gradlew bootRun` (serves :8080) needs a reachable PostgreSQL. See `docs/api-contract.md` for the
endpoint contract and the repo memory note for the local-DB port caveat on this dev machine.
